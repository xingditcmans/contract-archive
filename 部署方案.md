## 1. 文件清单与打包

### 1.1 项目核心文件

| 文件/目录 | 用途 | 是否必须 |
|-----------|------|:--------:|
| `backend/` | FastAPI 后端源码（Python） | ✅ |
| `backend/app/` | 核心应用代码 | ✅ |
| `backend/requirements.txt` | Python 依赖清单 | ✅ |
| `frontend/dist/` | 前端构建产物（已打包的 HTML/JS/CSS） | ✅ |
| `.env.example` | 环境变量模板 | ✅ |
| `start.sh` | Linux 一键启动脚本 | ✅ Linux |
| `stop.sh` | Linux 停止脚本 | ✅ Linux |
| `deploy.sh` | Linux 一键部署脚本（含 systemd） | ✅ Linux |
| `contract-archive.service` | systemd 服务配置模板 | ✅ Linux |
| `Dockerfile` | Docker 镜像构建文件 | ✅ Docker |
| `docker-compose.yml` | Docker 编排配置 | ✅ Docker |
| `.dockerignore` | Docker 构建排除规则 | ✅ Docker |
| `start.bat` | Windows 一键启动脚本 | ✅ Windows |
| `stop.bat` | Windows 停止脚本 | ✅ Windows |
| `.env` | 实际环境变量（含密钥，**不要打包**） | ❌ 部署时创建 |

### 1.2 打包命令

在项目根目录执行（排除冗余，压缩包约 600KB）：

```bash
# --- Windows (Git Bash) ---
cd /path/to/contract-archive
tar --exclude="backend/venv" --exclude="frontend/node_modules" \
    --exclude="__pycache__" --exclude="*.pyc" --exclude=".git" \
    --exclude="*.log" --exclude="backend/data/*" --exclude="backend/uploads/*" \
    -czf /path/to/contract-archive.tar.gz .

# --- Linux ---
cd /opt/contract-archive
tar --exclude="backend/venv" --exclude="frontend/node_modules" \
    --exclude="__pycache__" --exclude="*.pyc" --exclude=".git" \
    --exclude="*.log" --exclude="backend/data/*" --exclude="backend/uploads/*" \
    -czf /tmp/contract-archive.tar.gz .
```

**⚠️ 排除了什么？**

| 排除内容 | 原因 | 服务器上怎么得到 |
|----------|------|:----------------:|
| `backend/venv/` | 本地 Python 虚拟环境（~139MB） | `deploy.sh` 自动创建 |
| `frontend/node_modules/` | 前端构建依赖（~113MB） | 不需要，`dist/` 已包含构建结果 |
| `backend/data/` | 本地数据库文件（演示数据） | 服务器上重新生成 |
| `backend/uploads/` | 本地上传文件（空的） | 服务器上重新生成 |
| `__pycache__` / `*.pyc` | Python 字节码缓存 | 运行时自动生成 |
| `.git/` | Git 版本历史 | 不需要 |
| `*.log` | 本地日志 | 服务器上重新生成 |
| `.env` | **⚠️ 含明文密码/密钥！绝对不能打包** | 部署时创建 |

### 1.3 上传到服务器

```bash
# 从本地电脑上传到阿里云 ECS
scp /path/to/contract-archive.tar.gz root@你的ECS公网IP:/opt/
```

## 2. Linux 部署（已验证）

### 2.1 环境要求

| 项目 | 要求 |
|------|------|
| 操作系统 | Ubuntu 18.04+ / Debian 10+ / CentOS 7+ |
| Python | 3.10+（建议 3.12） |
| 内存 | ≥ 1GB（建议 2GB，OCR 识别较吃内存） |
| 磁盘 | ≥ 10GB |
| 端口 | 8000（可配置） |

### 2.2 快速部署（一键脚本）

```bash
# 1. 连接服务器
ssh root@你的ECS公网IP

# 2. 上传压缩包（在本地电脑执行）
# scp /path/to/contract-archive.tar.gz root@你的ECS公网IP:/opt/

# 3. 解压
cd /opt
mkdir -p contract-archive
tar -xzf contract-archive.tar.gz -C contract-archive/
cd contract-archive

# 4. 一键部署
chmod +x deploy.sh start.sh stop.sh
./deploy.sh

# 5. 等待部署完成，浏览器访问
# http://你的ECS公网IP:8000
```

### 2.3 分步部署（手动模式，便于排错）

如果一键部署失败，可按以下步骤手动执行。

#### 2.3.1 解压文件

```bash
cd /opt
mkdir -p contract-archive
tar -xzf contract-archive.tar.gz -C contract-archive/
cd contract-archive
ls -la
# 确认看到: backend/ frontend/ deploy.sh start.sh .env.example Dockerfile ...
```

#### 2.3.2 安装系统依赖

```bash
# Ubuntu / Debian
apt-get update
apt-get install -y python3 python3-venv python3-pip curl lsof \
    tesseract-ocr tesseract-ocr-chi-sim

# CentOS / RHEL
yum install -y python3 python3-pip curl lsof \
    tesseract tesseract-langpack-chi-sim
```

> **⚠️ 踩坑记录 — Tesseract OCR**
>
> 如果缺少 `tesseract-ocr` 或 `tesseract-ocr-chi-sim`，PDF 扫描件 OCR 识别会失败。
> 检查是否安装成功：
> ```bash
> tesseract --version           # 检查主程序
> tesseract --list-langs        # 检查语言包（应有 chi_sim）
> python3 -c "import pytesseract; print(pytesseract.__version__)"  # 检查 Python 绑定
> ```

#### 2.3.3 创建 Python 虚拟环境

```bash
cd /opt/contract-archive/backend
python3 -m venv venv
source venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

#### 2.3.4 配置环境变量

```bash
cd /opt/contract-archive/backend
cp ../.env.example .env

# 生成随机密钥
SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")
sed -i "s/^SECRET_KEY=.*/SECRET_KEY=$SECRET_KEY/" .env

# 设为生产模式
sed -i "s/^ENVIRONMENT=.*/ENVIRONMENT=production/" .env

# ⚠️ 关键修复：清除 +aiosqlite（如果有）
# 原因：database.py 使用同步 create_engine，URL 不能含 +aiosqlite
# 否则报错：greenlet_spawn has not been called
sed -i 's/+aiosqlite//' .env

# 验证配置文件
cat .env
```

**`.env` 正确示例：**
```
SECRET_KEY=a1b2c3d4e5f6...（64位随机十六进制）
DATABASE_URL=sqlite:///data/contracts.db   # ❌ 不能是 sqlite+aiosqlite:///
ENVIRONMENT=production
```

#### 2.3.5 初始化目录

```bash
cd /opt/contract-archive/backend
mkdir -p data uploads
```

#### 2.3.6 启动服务（前台测试）

```bash
cd /opt/contract-archive/backend
source venv/bin/activate
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

看到 `Uvicorn running on http://0.0.0.0:8000` 表示成功。

#### 2.3.7 开放端口

**阿里云安全组配置：**
- 登录阿里云控制台 → ECS 实例 → 安全组 → 配置规则
- 入方向 → 添加规则：
  - 协议类型：TCP
  - 端口范围：8000
  - 授权对象：0.0.0.0/0
  - 描述：合同归档系统

> **⚠️ 踩坑记录 — 端口未开放**
>
> 服务启动后浏览器访问 502，原因通常是阿里云安全组没放行 8000 端口。
> 安全组配置完成后无需重启服务，立即生效。

#### 2.3.8 注册 systemd 服务（开机自启）

创建 `/etc/systemd/system/contract-archive.service`：

```ini
[Unit]
Description=合同归档管理系统
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/contract-archive/backend
ExecStart=/opt/contract-archive/backend/venv/bin/python \
    -m uvicorn app.main:app --host 0.0.0.0 --port 8000
Restart=always
RestartSec=5
Environment=PATH=/opt/contract-archive/backend/venv/bin:/usr/local/bin:/usr/bin:/bin

# 安全加固
NoNewPrivileges=yes
PrivateTmp=yes

[Install]
WantedBy=multi-user.target
```

```bash
# 启用并启动
systemctl daemon-reload
systemctl enable contract-archive
systemctl start contract-archive

# 查看状态
systemctl status contract-archive
journalctl -u contract-archive -f    # 实时日志
```

### 2.4 管理命令

```bash
# 启动 / 停止 / 重启
systemctl start contract-archive
systemctl stop contract-archive
systemctl restart contract-archive

# 查看状态
systemctl status contract-archive

# 查看日志
journalctl -u contract-archive -f --no-hostname

# 手动启动（前台调试）
cd /opt/contract-archive/backend && source venv/bin/activate && \
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

## 3. Docker 部署（已验证 ✅）

> ✅ 2026-06-22 在阿里云 ECS（Ubuntu 22.04）上验证通过
> Docker 版本: 24.x | docker-compose 版本: 1.29.2（旧版兼容性问题见5.8）

### 3.1 环境要求

| 项目 | 要求 |
|------|------|
| Docker | 20.10+（推荐 24.x） |
| 内存 | ≥ 1GB |
| 磁盘 | ≥ 5GB |

> **⚠️ 国内服务器必须先配镜像加速器**，否则 Docker Hub 拉不下来（详见 5.7 节）

### 3.2 文件清单

| 文件 | 用途 |
|------|------|
| `Dockerfile` | 基于 python:3.12-slim，内置 tesseract-ocr + 中文语言包 |
| `docker-compose.yml` | 数据卷挂载 data/ 和 uploads/，端口映射 |
| `.dockerignore` | 排除 venv/node_modules/.git 等 |

### 3.3 快速启动（推荐 — docker run）

由于旧版 docker-compose（≤1.x）存在 `ContainerConfig` 兼容性 bug，推荐直接用 `docker run`：

```bash
# 1. 解压项目（若从压缩包部署）
cd /opt
mkdir -p contract-archive
tar -xzf contract-archive.tar.gz -C contract-archive/
cd contract-archive

# 2. 构建镜像
docker build -t contract-archive .

# 3. 启动容器（一条命令）
docker run -d \
  --name contract-archive \
  -p 8000:8000 \
  -e ENVIRONMENT=production \
  -e SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))") \
  -v contract_data:/app/backend/data \
  -v contract_uploads:/app/backend/uploads \
  --restart unless-stopped \
  contract-archive

# 4. 检查
docker ps
curl http://localhost:8000/health
```

### 3.4 使用 docker-compose（Docker Compose v2+ 可用）

```bash
# 配置 SECRET_KEY（取消注释并填入随机密钥）
cd /opt/contract-archive
sed -i "s|# - SECRET_KEY=| - SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")|" docker-compose.yml

# 启动
docker-compose up -d
```

### 3.5 管理命令

```bash
# 查看运行状态
docker ps
docker stats contract-archive    # 实时资源占用

# 查看日志
docker logs contract-archive -f --tail 50

# 进入容器
docker exec -it contract-archive bash

# 停止
docker stop contract-archive

# 启动
docker start contract-archive

# 删除容器（数据不丢失，因为 volumes 持久化）
docker rm -f contract-archive

# 删除旧镜像（重新构建前需要）
docker rmi contract-archive
```

### 3.6 数据持久化

Docker 模式下数据库和上传文件通过 volume 挂载到宿主机：

| 容器内路径 | 卷名 | 宿主机位置 | 说明 |
|------------|------|-----------|------|
| `/app/backend/data/` | `contract_data` | `/var/lib/docker/volumes/contract_data/` | SQLite 数据库 |
| `/app/backend/uploads/` | `contract_uploads` | `/var/lib/docker/volumes/contract_uploads/` | 上传附件 |

**删除容器不会丢失数据**，但如需彻底清理（重新开始），需手动删除 volume：

```bash
docker volume rm contract_data contract_uploads
```

### 3.7 从 Docker 切换到 Linux 部署（如需）

如果 Docker 跑着但想换回直接部署，备份数据库再重建即可：

```bash
# 1. 从容器里拷出数据库文件
docker cp contract-archive:/app/backend/data/contracts.db .
docker cp -a contract-archive:/app/backend/uploads .

# 2. 停掉 Docker
docker stop contract-archive

# 3. 按第2节 Linux 部署步骤走，把备份的数据库拷回 backend/data/
```

## 5. 常见问题与排坑

### 5.1 🐛 deploy.sh: root 用户检查拒绝

**现象：**
```
[错误] 请不要用 sudo 运行此脚本，直接 ./deploy.sh 即可
```

**原因：** ECS 默认使用 root 登录，旧版 `deploy.sh` 拒绝 root 用户运行。

**修复（v1.0→v1.1）：** 取消 root 限制，兼容 root 和非 root 两种场景。

**手动绕过（旧脚本）：**
```bash
sed -i 's/exit 1/# exit 1/' deploy.sh
./deploy.sh
```

### 5.2 🐛 deploy.sh: 复制文件报 "same file"

**现象：**
```
cp: "contract-archive" and "contract-archive" are the same file
```

**原因：** 压缩包解压目录与 `deploy.sh` 的 `INSTALL_DIR` 默认值相同（都是 `/opt/contract-archive`），脚本试图复制到自己。

**修复（v1.0→v1.1）：** 增加判断，源码目录与安装目录相同时跳过复制步骤。

**手动绕过：** 无需跳过，直接手动执行后续步骤。

### 5.3 🐛 greenlet_spawn has not been called

**现象：**
```
sqlalchemy.exc.InvalidRequestError: greenlet_spawn has not been called
```

**原因：** `.env` 中 `DATABASE_URL=sqlite+aiosqlite:///data/contracts.db` 包含了 `+aiosqlite`（异步驱动），但 `database.py` 使用同步的 `create_engine`，不兼容。

**修复方法：**

```bash
# 在 ECS 上执行
cd /opt/contract-archive/backend
sed -i 's/+aiosqlite//' .env
cat .env   # 确认 DATABASE_URL=sqlite:///data/contracts.db
```

**深层修复（v1.0→v1.1）：** `deploy.sh` 中自动执行 `sed -i 's/+aiosqlite//'` 清理。

### 5.4 🐛 OCR 超时 / 识别失败

**现象：** 上传 PDF 后提示「OCR 处理超时」或「未能自动识别关键字段」。

**原因分析：**

| 可能原因 | 判断方法 | 解决方法 |
|----------|----------|----------|
| Tesseract-OCR 未安装 | `tesseract --version` 报错 | `apt-get install tesseract-ocr tesseract-ocr-chi-sim` |
| 中文语言包缺失 | `tesseract --list-langs` 无 chi_sim | 安装 `tesseract-ocr-chi-sim` |
| ECS 内存不足（<1GB） | `free -h` 查看 | 升级 ECS 配置或缩小 PDF 页数 |
| PDF 页数过多（>10页） | 代码限制最多 OCR 前10页 | 拆分 PDF 后重试 |

**验证 OCR 环境：**

```bash
source /opt/contract-archive/backend/venv/bin/activate
python3 -c "
import pytesseract
print('pytesseract OK:', pytesseract.__version__)
print('Tesseract path:', pytesseract.pytesseract.tesseract_cmd)
import fitz; print('PyMuPDF OK')
import pdfplumber; print('pdfplumber OK')
"
```

### 5.5 🐛 502 Bad Gateway

**现象：** 浏览器访问 `http://公网IP:8000` 显示 502。

**原因：** 阿里云安全组未放行 8000 端口。

**解决方法：**
1. 登录阿里云控制台
2. ECS 实例 → 安全组 → 配置规则 → 入方向
3. 添加规则：TCP/8000，授权对象 `0.0.0.0/0`

### 5.6 🐛 阿里云 ECS 的 IP 显示问题

**现象：** 服务跑在 `0.0.0.0:8000`，但浏览器访问 `http://公网IP:8000` 不通。

**确认方法：**

```bash
# 检查服务是否在监听
ss -tlnp | grep 8000

# 检查本地是否正常
curl http://localhost:8000/health

# 检查公网 IP
curl ifconfig.me
```

### 5.7 🐛 Docker: docker pull 超时 / 镜像拉不下来（国内网络）

**现象：**
```
failed to resolve reference "docker.io/library/python:3.12-slim": not found
```
或长时间卡在 `Waiting` 后超时。

**原因：** Docker Hub 在国内被限制，国内服务器无法直接拉取。

**解决方法 —— 配置镜像加速器：**

```bash
# 方法A：阿里云专属加速器（推荐，阿里云 ECS 用户）
# 1. 登录 https://cr.console.aliyun.com → 镜像中心 → 镜像加速器
# 2. 复制你的专属加速地址（格式：xxxx.mirror.aliyuncs.com）
cat > /etc/docker/daemon.json << 'EOF'
{
  "registry-mirrors": ["https://你的专属地址.mirror.aliyuncs.com"]
}
EOF
systemctl restart docker

# 方法B：DaoCloud / 网易 / 中科大公共加速器（阿里云专属不可用时）
cat > /etc/docker/daemon.json << 'EOF'
{
  "registry-mirrors": [
    "https://docker.m.daocloud.io",
    "https://hub-mirror.c.163.com",
    "https://docker.mirrors.ustc.edu.cn"
  ]
}
EOF
systemctl restart docker

# 验证
docker pull hello-world
```

> **踩坑记录：** 阿里云专属加速器能拉 hello-world 等热门镜像，
> 但某些 Python tag（如 `python:3.13-slim-bookworm`）可能没有缓存，
> 会报 `not found`。改用 `python:3.12-slim` 即可（阿里云有缓存）。

### 5.8 🐛 Docker: docker-compose 报 'ContainerConfig' KeyError

**现象：**
```
File ".../service.py", line 1579, in get_container_data_volumes
  container.image_config['ContainerConfig'].get('Volumes') or {}
KeyError: 'ContainerConfig'
```

**原因：** docker-compose 1.x（如 v1.29.2）与新版 Docker Engine（24.x）不兼容，
旧容器镜像的元数据结构变了。

**解决方法（三选一）：**

```bash
# 方案A：换用 docker run（推荐，最简单）
# 先删除旧容器 + 旧镜像
docker rm -f contract-archive 2>/dev/null
docker rmi -f opt_contract-archive 2>/dev/null

# 然后用 docker run 启动（见 3.3 节）
docker run -d \
  --name contract-archive \
  -p 8000:8000 \
  -e ENVIRONMENT=production \
  -e SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))") \
  -v contract_data:/app/backend/data \
  -v contract_uploads:/app/backend/uploads \
  --restart unless-stopped \
  contract-archive

# 方案B：升级 docker-compose
curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" \
  -o /usr/local/bin/docker-compose
chmod +x /usr/local/bin/docker-compose

# 方案C：跳过旧容器数据卷缓存
docker-compose up -d --force-recreate
```

### 5.9 🐛 Docker: 生产环境 SECRET_KEY 未设置

**现象：**
```
ValueError: ⚠️ 生产环境必须设置 SECRET_KEY 环境变量！
```

**原因：** 容器内 `ENVIRONMENT=production`，但没有传入 `SECRET_KEY` 环境变量。
`config.py` 的生产环境验证会抛异常。

**解决方法：**

```bash
# docker run 时通过 -e 传入（见 3.3 节）
-e SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")

# docker-compose 时在 environment 中配置（见 3.4 节）
sed -i "s|# - SECRET_KEY=| - SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")|" docker-compose.yml
```

### 5.10 🐛 Docker: apt-get install libgl1-mesa-glx 找不到

**现象：**
```
E: Unable to locate package libgl1-mesa-glx
```

**原因：** Debian Bookworm 及以上版本中，`libgl1-mesa-glx` 已更名为 `libgl1`，
`python:3.12-slim` 基于 Bookworm。

**修复：** Dockerfile 中改包名 `libgl1-mesa-glx` → `libgl1`，已在最新版本中修复。
